Project

FreeRTOS System Monitor

Seven preemptively-scheduled tasks on an STM32H563 Nucleo, with a live CLI to watch the scheduler work

FreeRTOS STM32H563 Cortex-M33 Embedded C Task Notifications Queues Stream Buffers Event Groups

Overview

A FreeRTOS demo running on the STM32H563ZI Nucleo board (Cortex-M33, 250 MHz) — no extra hardware needed beyond the board and a USB cable. Seven tasks run at the same time: they blink a heartbeat LED, measure how precisely the scheduler wakes tasks up, watch each other's health, and expose all of it through a command line over the USB serial port.

Try it

  1. Import the project in STM32CubeIDE, build, and flash.
  2. Open a serial terminal (PuTTY, Tera Term...) on the ST-LINK COM port, 115200 baud.
  3. Type help.
h563> stats        <- table of all tasks: who is running, using CPU, stack left
h563> latency      <- how precisely tasks wake up (measured in microseconds)
h563> mode load    <- make the worker task burn CPU, then check stats again

Or just press the blue USER button — it cycles through the modes. LEDs: green blinking = system alive · yellow = a special mode is active · red = a task stopped responding (or a fatal error).

The tasks

TaskPriorityWhat it does
latency5 (highest)Wakes up exactly every 1 ms and measures how late it actually woke up, using the CPU's cycle counter — the "how real-time are we?" probe.
button4Sleeps until you press the blue button, then switches the system mode.
superv3The watchdog. Every task must report in every 3 seconds — if one goes silent, the red LED turns on.
cli2Reads what you type in the terminal and runs your commands.
logger2The only task allowed to print. Everyone else hands it messages, so output never gets scrambled.
heartbeat1Blinks the green LED every 500 ms. If it stops blinking, the scheduler is in trouble.
worker1 (lowest)Normally idle. In LOAD mode it does heavy math on purpose, so CPU usage shift is visible in stats while higher-priority tasks stay unaffected.

Higher priority always wins: when the worker is burning CPU, the latency task still wakes up on time because it outranks the worker. That's the whole point of an RTOS, and it's visible live with latency.

FreeRTOS features used, where, and why

FeatureWhereWhy this one
Task notificationbutton ISR → button taskFastest, lightest way to signal one task from an interrupt — cheaper than a queue or semaphore.
Queue (of structs)all tasks → logger; button → workerCarries whole messages safely between tasks. One owner of the UART fed by a queue is the gatekeeper pattern — it avoids a mutex around a slow peripheral.
Stream bufferUART RX interrupt → cli taskMade exactly for byte streams with one reader — a perfect fit for characters arriving one at a time.
Event grouptasks → supervLets the watchdog wait for all health bits at once with a single timeout, instead of one semaphore per task.
vTaskDelayUntillatency taskGives a fixed wake-up rhythm with no drift, unlike vTaskDelay which drifts by however long the work took.
Priorities + preemptioneverywhereThe scheduler always runs the highest-priority ready task — proven live by the worker-vs-latency demo.
Run-time statisticsstats commandPer-task CPU % and unused stack, driven by the CPU cycle counter.
Stack overflow + malloc hooksapp_main.cA blown stack or exhausted heap turns on the red LED with a message instead of a silent crash.

A note on the kernel

ST doesn't offer FreeRTOS for the H5 series in CubeMX, so the kernel (V11.3.0) is included as source in Middlewares/ and integrated by hand: the Cortex-M33 port owns the SysTick interrupt, and the HAL gets its millisecond tick from the DWT cycle counter instead.

CubeMX doesn't know FreeRTOS is there. After regenerating code from the .ioc file, two hand-written pieces have to be restored: the SVC_Handler, PendSV_Handler, and SysTick_Handler in stm32h5xx_it.c (the FreeRTOS port provides these), and the BSP_PB_Callback in main.c (lives in App/Src/control.c).

Where the code lives

App/Src/ the interesting part - one file per feature area app_main.c startup, kernel hooks, HAL tick retarget logger.c gatekeeper logger (queue) cli.c command line (stream buffer) control.c button, modes, worker, heartbeat monitor.c latency probe + watchdog (event group) Core/ CubeMX-generated board setup + FreeRTOSConfig.h Middlewares/ FreeRTOS kernel source (unmodified) Drivers/ ST HAL + Nucleo board support (generated)

What I Learned

  • Hand-integrating a FreeRTOS port ST doesn't officially support in CubeMX, including retargeting the HAL tick to the DWT cycle counter
  • Picking the right primitive for each job — task notification vs. queue vs. stream buffer vs. event group — instead of reaching for one tool everywhere
  • The gatekeeper pattern for shared peripherals: one task owns the UART, everyone else sends it messages through a queue
  • vTaskDelayUntil vs. vTaskDelay: fixed-rhythm wake-ups versus wake-ups that drift with execution time
  • Making failures visible — stack overflow and malloc-failed hooks that light an LED instead of failing silently
